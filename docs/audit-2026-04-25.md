# Auditoría técnica — 2026-04-25

## Resumen ejecutivo

El proyecto está en una **base funcional buena**: compila, el frontend builda, el backend pasa `cargo check`, hay una separación razonable entre Tauri/Rust y React/TypeScript, y se han incorporado capacidades potentes para VirtualDJ (playlists, waveforms, settings, mappers, pads, carpetas externas, validaciones de escritura).

Aun así, la auditoría detecta una **deuda técnica importante en tres frentes**:

1. **Flujo de datos y UX de recarga**: varias acciones mutantes fuerzan recargas globales costosas y en algunos casos rompen la navegación actual.
2. **Escalabilidad/mantenibilidad del frontend**: hay componentes y páginas que concentran demasiada lógica (`SongTable`, `Songs`, `Pads`, `Mappers`, `App`).
3. **Calidad operativa del proyecto**: faltan tests, CI, logging estructurado y documentación actualizada respecto al estado real del repo.

## Validaciones ejecutadas en esta auditoría

- `tsc --noEmit && oxlint src/` ✅
- `cargo check` ✅
- `bun run build` ✅
  - salida verificada en `dist/`
  - bundle JS principal: ~402 kB

## Fortalezas verificadas

### 1. Buen reparto base entre frontend y backend

- `src/lib/api.ts`
- `src-tauri/src/lib.rs`
- `src-tauri/src/commands/*`

La app mantiene una separación clara: el frontend usa wrappers IPC y el backend concentra filesystem/parsing/operaciones de VirtualDJ.

### 2. Escrituras de `database.xml` mejor protegidas que en una app Tauri promedio

- `src-tauri/src/database/parser.rs`
- `src-tauri/src/commands/database.rs`
- `src-tauri/src/commands/files.rs`

Existe validación previa y reparseo posterior (`write_database_checked`), además de backups antes de mutar. Es una muy buena decisión dado lo delicado del XML de VirtualDJ.

### 3. Trabajo de waveform bastante sólido

- `src/components/WaveformPreview.tsx`
- `src-tauri/src/commands/waveforms.rs`

Hay cola en frontend, caché en memoria/disco, y límite de concurrencia en backend mediante `Semaphore`.

### 4. Cobertura funcional amplia para una herramienta interna

- `Songs`, `Duplicates`, `MissingFiles`, `OrphanFiles`, `BatchOperations`, `Configs`, `Playlists`, `Pads`, `Mappers`

La app ya cubre bastantes flujos reales del usuario VirtualDJ y no se siente como un simple visor XML.

## Hallazgos críticos

### C1. `reload()` resetea la navegación del usuario a `dashboard`

Evidencia:

- `src/App.tsx`

`reload()` llama a `loadFromFolder(state.vdjFolder)` y `loadFromFolder()` hace siempre:

- `page: "dashboard"`

Impacto:

- Editar tags en `SongTable`
- operar desde `Duplicates`
- guardar desde páginas que llaman `reload()`

puede mandar al usuario de vuelta al dashboard aunque estuviera trabajando en otra vista.

Riesgo:

- UX frustrante
- sensación de “la vista salta sola”
- pérdida de contexto visual

Recomendación:

- separar `loadFromFolder()` de `refreshCurrentData()`
- preservar `page` actual durante recargas internas
- sólo navegar a `dashboard` en la carga inicial/manual de carpeta

### C2. Ediciones unitarias hacen recarga global completa de base y estadísticas

Evidencia:

- `src/components/SongTable.tsx`
- `src/App.tsx`

Tras editar un tag, estrellas o color se hace:

- `updateSongTags(...)`
- `reload()`

Eso vuelve a cargar canciones + estadísticas completas.

Impacto:

- coste innecesario por cada edición puntual
- peor latencia percibida
- mayor presión sobre IPC y parseo XML

Recomendación:

- aplicar actualización optimista/local de la fila afectada
- invalidar stats sólo cuando realmente cambien
- ofrecer `refreshStats()` separado si hace falta

### C3. Operaciones batch de tags escriben N veces el XML completo

Evidencia:

- `src/pages/BatchOperations.tsx`
- `src-tauri/src/commands/database.rs`

En modo `tag`, el frontend itera canción por canción llamando `updateSongTags()` para cada índice.
Cada llamada:

- parsea DB
- muta una sola entrada
- crea backup
- escribe XML
- revalida

Impacto:

- coste explosivo para lotes grandes
- muchas escrituras y backups redundantes
- más riesgo operativo

Recomendación:

- crear comando backend batch para aplicar patch a múltiples índices en una sola escritura
- reutilizar el patrón de `save_database` o un `update_song_tags_batch`

## Hallazgos altos

### H1. `SongTable.tsx` es un “mega-componente” con demasiadas responsabilidades

Evidencia:

- `src/components/SongTable.tsx`

Gestiona:

- definición de columnas
- sorting/search
- inline editing
- ratings
- color picker
- audio preview
- virtualización
- contexto de columnas
- mini table exportada en el mismo archivo
- feedback de cola de waveform

Impacto:

- difícil de testear
- difícil de evolucionar
- alto riesgo de regresiones

Recomendación:

extraer al menos:

- `songTableColumns.tsx`
- `useSongTablePreferences.ts`
- `useSongInlineEditing.ts`
- `SongColorPicker.tsx`
- `SongPlaybackCell.tsx`
- `SongMiniTable.tsx` en archivo propio

### H2. `Songs.tsx` mezcla dos productos en una sola pantalla

Evidencia:

- `src/pages/Songs.tsx`

La página combina:

- browser por carpetas
- browser por playlists
- importación de playlist externa
- escaneo de música externa
- síntesis de canciones no presentes en DB
- detalle de tema seleccionado

Impacto:

- lógica duplicada con `Playlists.tsx`
- mayor superficie de bugs
- más difícil definir UX consistente

Recomendación:

- dejar `Songs` enfocada a biblioteca/carpetas
- consolidar toda la UX de playlists en `Playlists.tsx`
- extraer helpers compartidos para `createExternalSong` y resolución path→song

### H3. Duplicación de lógica de “song externa”

Evidencia:

- `src/pages/Songs.tsx`
- `src/pages/Playlists.tsx`

Se construyen objetos `SongSummary` sintéticos en varios sitios.

Impacto:

- drift entre vistas
- cambios futuros más caros

Recomendación:

- crear helper único tipo `buildExternalSongSummary(filePath, index, reason)`

### H4. `Pads` y `Mappers` crecieron muy rápido hacia componentes monolíticos

Evidencia:

- `src/pages/Pads.tsx`
- `src/pages/Mappers.tsx`

Ambas páginas ya cargan:

- árbol de archivos
- carga raw + estructurada
- estado dirty/backup
- edición visual propia

Impacto:

- bien para prototipo funcional
- caro de mantener a medio plazo

Recomendación:

extraer:

- hooks de carga/guardado (`useVdjResourceFiles`, `useVdjMapperEditor`, `useVdjXmlDocumentEditor`)
- componentes comunes de editor lateral y header de recurso

### H5. Documentación principal quedó desactualizada respecto al proyecto real

Evidencia:

- `README.md`
- `docs/architecture.md`
- `docs/tech-debt.md`

La documentación aún no refleja bien:

- `Playlists`
- `Pads`
- `Mappers`
- `Configs` centrado en `settings.xml`
- parseo estructurado de `.vdjmap` y `.vdjpad`
- integración de carpetas musicales externas

Impacto:

- onboarding engañoso
- arquitectura percibida menor que la real

Recomendación:

actualizar README + architecture + tech-debt tras esta auditoría

## Hallazgos medios

### M1. No hay tests en frontend ni backend

Evidencia:

- búsqueda de `*.test.*` / `*.spec.*`: sin resultados

#### Impacto

- alto riesgo de regressions
- especialmente delicado en parser XML, playlists, pads/mappers y batch ops

Prioridad sugerida:

1. tests Rust para parser/playlist/configs
2. tests TS para utilidades API/helpers
3. tests React para `SongTable`, `Configs`, `Playlists`

### M2. No hay CI/CD definido

Evidencia:

- no se detecta pipeline en el repo

Recomendación:

GitHub Actions mínimo con:

- `bun run check`
- `bun run build`
- `cargo check`
- opcional `cargo clippy`

### M3. Falta logging estructurado en backend

Evidencia:

- `src-tauri/src/commands/*`
- no se observa `tracing` ni `log`

Impacto:

- diagnóstico difícil en fallos reales del filesystem/XML/audio

### M4. Lógica de backup duplicada en varios comandos Rust

Evidencia:

- `src-tauri/src/commands/database.rs`
- `src-tauri/src/commands/files.rs`
- `src-tauri/src/commands/configs.rs`

Problema:

- hay múltiples formas de crear backups
- algunas ignoran errores con `.ok()`
- no hay helper central para DB/config/resources

Recomendación:

- centralizar helpers de backup
- no silenciar errores de backup cuando el archivo es crítico

### M5. `scan_music_folder()` puede ser caro para raíces grandes

Evidencia:

- `src-tauri/src/commands/files.rs`
- `src/pages/Songs.tsx`

Se hace walk recursivo completo de carpetas; desde frontend puede dispararse por cambios de selección/modo.

Riesgo:

- lentitud con bibliotecas grandes
- coste repetido

Recomendación:

- caché de escaneo por raíz
- fingerprint por mtime si procede
- limitar frecuencia / debounce desde frontend

### M6. `Playlists.tsx` muestra bien el árbol, pero no tiene acciones operativas todavía

Evidencia:

- `src/pages/Playlists.tsx`

Faltan cosas esperables para una vista madura:

- crear/renombrar/mover playlists
- operar sobre nodos carpeta
- importar/exportar desde la propia página

### M7. `list_vdj_config_files()` es útil, pero demasiado genérico para crecer sin control

Evidencia:

- `src-tauri/src/commands/configs.rs`

Hoy mezcla settings, mappings, pads y otros recursos editables bajo un mismo comando genérico.

Riesgo:

- con el tiempo puede convertirse en “cajón de sastre”

Recomendación:

- introducir comandos por dominio si la app sigue creciendo:
  - `list_vdj_mappers`
  - `list_vdj_pads`
  - `list_vdj_playlists_resources`

## Riesgos de producto / UX

### UX1. El usuario puede romper XML válidos en `Pads`/`Mappers` si no hay validaciones semánticas

Los editores actuales son ya mucho mejores que raw text, pero aún no validan reglas de dominio de VirtualDJ.

Ejemplo:

- nombres de nodo arbitrarios en `Pads`
- bindings vacíos o duplicados en `Mappers`

Siguiente paso recomendado:

- validadores específicos de `.vdjmap`
- vista semántica de pad pages/slots/acciones

### UX2. La app mezcla patrones visuales distintos entre páginas nuevas y antiguas

`Configs`, `Playlists`, `Pads`, `Mappers` tienen un lenguaje de interfaz más reciente/estructurado que algunas vistas clásicas.

Oportunidad:

- unificar layout de sidebar + panel detalle + acciones + backup badge + dirty state

## Veredicto final

El proyecto está en un punto **muy prometedor y ya útil**, pero necesita una pasada de consolidación.

No veo una crisis de arquitectura, pero sí una señal clara de crecimiento rápido: el producto ya tiene más capacidades de las que su estructura y documentación acompañan cómodamente.

### Nota final

Si hubiera que resumir la auditoría en una frase:

> **La app ya funciona como herramienta seria de VirtualDJ, pero ahora toca entrar en fase de endurecimiento: menos recargas globales, menos componentes monolíticos, más tests, más CI y documentación alineada con la realidad.**
