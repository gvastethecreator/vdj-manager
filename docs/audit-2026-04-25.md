# Auditoria tecnica y de producto - VDJ Database Manager

Fecha: 2026-04-25

## Alcance

Esta auditoria revisa el estado actual de `D:\DEV\vdj-manager`, una app de escritorio Rust + Tauri + React para leer, analizar y modificar recursos de VirtualDJ.

Se revisaron:

- documentacion local en `docs/` y `docs/vdj-docs/`;
- documentacion oficial publica de VirtualDJ;
- backend Rust/Tauri en `src-tauri/src/`;
- frontend React en `src/`;
- scripts, configuracion y empaquetado;
- validaciones locales.

## Fuentes VirtualDJ usadas

- VDJPedia `VDJ_database`: `https://www.virtualdj.com/wiki/VDJ_database.html`
- VDJPedia `Developers`: `https://es.virtualdj.com/wiki/Developers`
- Documentacion local: `docs/vdj-docs/**`

Punto central de la fuente oficial: VirtualDJ documenta `database.xml` para lectura externa, pero desaconseja modificarlo desde apps externas y no ofrece soporte si la base queda corrupta. Eso vuelve criticas todas las rutas de escritura de este proyecto: editar tags, relocalizar, mover, renombrar, borrar, settings, mappers y pads.

## Documentacion local: gap de dominio

La carpeta `docs/vdj-docs/` es rica, pero esta enfocada mayormente en skins, VDJScript, opciones y referencias de interfaz. Para este proyecto falta una referencia local especifica de base de datos:

- esquema completo `VirtualDJ_Database` / `Song` / `Tags` / `Infos` / `Scan` / `Poi`;
- advertencia oficial de escritura externa;
- reglas operativas para guardar sin corrupcion;
- convenciones de `FilePath`, `FileSize`, `Scan/@Bpm` y fechas/campos de `Infos`;
- diferencia entre `Tags/@Bpm` y `Scan/@Bpm`;
- politica de compatibilidad con `VirtualDJ Database v6.xml` pre-v8.

Esto importa porque el proyecto ya modifica `database.xml`, pero la documentacion local no define todavia el contrato de seguridad de esas escrituras.

## Hallazgos criticos restantes

### C1. El modelo de escritura no preserva XML desconocido con garantia fuerte

Evidencia:

- `VdjDatabase`, `Song`, `Tags`, `Infos`, `Scan`, `Poi` modelan campos conocidos con serde.
- `write_database()` serializa desde structs, no desde el arbol original.
- `src-tauri/tests/database_write.rs` documenta el gap con `database_unknown_attrs.xml`.

Riesgo:

- Si VirtualDJ agrega atributos nuevos o el usuario tiene metadata no modelada, se pueden perder al guardar.
- La documentacion oficial lista un schema basico, pero no garantiza que no existan variantes o atributos nuevos.

Recomendacion:

- para edicion de tags, usar estrategia patch-in-place sobre XML preservando nodos/atributos desconocidos;
- o agregar `other_attributes` a modelos criticos y tests de preservacion.

### C2. Operaciones de archivo todavia no son completamente transaccionales frente a la DB

Evidencia:

- `rename_file_op` renombra archivo fisico y despues escribe `database.xml`.
- `move_files_op` mueve/copia multiples archivos y despues escribe `database.xml`.
- El fallo de `remove_file` cross-drive ya no se ignora, pero aun falta una transaccion completa con rollback por item.

Riesgo:

- Si el write de DB falla despues de mover archivos, disco y DB pueden quedar divergentes.

Recomendacion:

- crear un plan de rollback por archivo antes de tocar disco;
- persistir resultados por item con estado `moved`, `copied`, `delete_failed`, `db_updated`, `db_failed`;
- mostrar una pantalla de recuperacion si el commit de DB falla.

## Hallazgos altos

### H1. `scan_music_folder` y busquedas similares siguen symlinks y silencian errores

Evidencia:

- `WalkDir::new(...).follow_links(true).filter_map(|e| e.ok())` en `files.rs` y `configs.rs`.

Riesgo:

- Escaneos grandes pueden cruzar carpetas inesperadas.
- Symlink loops o permisos quedan invisibles para el usuario.
- La UX solo ve resultados incompletos, sin saber que hubo errores.

Recomendacion:

- desactivar `follow_links(true)` por defecto o hacerlo configurable;
- devolver `ScanReport { files, skipped, errors }`;
- agregar limites y cancelacion para escaneos largos.

### H2. La superficie Tauri permite FS plugin amplio aunque el backend ya hace casi todo

Evidencia:

- `src-tauri/capabilities/default.json` habilita permisos FS de lectura, escritura, mkdir, rename, copy y remove.

Riesgo:

- Aumenta superficie de seguridad del WebView.
- La app ya canaliza operaciones sensibles por comandos Rust, que son mas auditables.

Recomendacion:

- reducir permisos FS del frontend a lo estrictamente necesario;
- preferir comandos Rust para mutaciones;
- documentar por que cada permiso sigue existiendo.

## Hallazgos medios

### M1. `SongTable.tsx` sigue siendo un mega-componente

Responsabilidades mezcladas:

- columnas;
- renderers;
- inline editing;
- rating;
- color picker;
- audio preview;
- localStorage de columnas;
- virtualizacion;
- mini table.

Recomendacion:

- extraer `songTableColumns.tsx`;
- extraer `useSongInlineEditing`;
- extraer `SongColorPicker`;
- extraer `SongMiniTable`;
- mover helpers de key/color a modulos puros testeables.

### M2. Duplicacion de canciones externas

Evidencia:

- `src/pages/Songs.tsx` crea `SongSummary` sinteticos para archivos externos.
- `src/pages/Playlists.tsx` tiene `createExternalSong`.

Recomendacion:

- crear `src/lib/songSummary.ts`;
- exponer `createExternalSongSummary(filePath, index, reason)`;
- cubrirlo con tests TS.

### M3. `Songs.tsx` mezcla biblioteca, carpetas externas y playlists

El modo playlist dentro de `Songs` duplica parte del producto que ya existe en `Playlists.tsx`.

Recomendacion:

- decidir frontera: `Songs` = biblioteca/carpetas; `Playlists` = listas e historial;
- mantener importacion de playlist externa solo en `Playlists`;
- compartir resolucion path -> song.

### M4. `list_vdj_config_files` puede convertirse en cajon de sastre

Evidencia:

- lista extensiones genericas y recorre toda la carpeta VirtualDJ.

Recomendacion:

- separar comandos por dominio: settings, mappers, pads, playlists;
- mantener el explorador generico como modo avanzado/read-only.

### M5. No hay logging backend estructurado

Recomendacion:

- agregar `tracing`;
- instrumentar comandos mutantes y escaneos largos;
- escribir logs locales con rotacion ligera.

## Hallazgos bajos

### L1. `lint` y `format` tienen el mismo comando

`package.json` define ambos como `oxlint --fix src/`. Esto confunde la diferencia entre verificar y formatear.

### L2. Bundle frontend no es critico, pero debe vigilarse

`416.57 kB` JS principal no es alarmante para Tauri, pero conviene medir cuando se agreguen editores mas pesados.

### L3. La documentacion `docs/vdj-docs` habla de Luna/skins

No esta mal conservarla si sirve como referencia VirtualDJ, pero hoy el README de esa carpeta no explica que el proyecto actual es un manager de base de datos, no un skin.

## Hallazgos ya resueltos frente a la auditoria previa

- `reload()` ya preserva la pagina actual mediante `targetPage`.
- Las ediciones inline de `SongTable` ya usan `patchSong` y no hacen reload global.
- El batch de tags ya usa `saveSongUpdates()` y una sola escritura.
- `bun run build` y `bun run tauri build` pasan.
- `database.xml` y recursos XML ahora usan escritura atomica por helper compartido.
- Los backups criticos ya no se silencian con `.ok()`.
- `update_song_tags` usa payload estructurado.
- Los errores de edicion inline suben a `setError`.
- Hay fixtures y tests Rust de parse/write/reparse y helpers de seguridad.
- Hay CI minima en `.github/workflows/ci.yml`.

## Prioridades recomendadas

### Sprint 1: seguridad de datos y guardrails

1. Preservar atributos/nodos XML desconocidos al escribir `database.xml`.
2. Agregar fixtures de `.m3u`, `.vdjplaylist`, `.vdjmap`, `.vdjpad` y `settings.xml`.
3. Cubrir comandos de configs/playlists con tests Rust.
4. Reducir permisos Tauri FS.

### Sprint 2: robustez operativa

1. Transacciones/rollback para rename, move y relocate.
2. `ScanReport` con errores y cancelacion para escaneos largos.
3. Logging backend con `tracing`.
4. Menor superficie de permisos Tauri FS.

### Sprint 3: mantenibilidad y UX

1. Partir `SongTable.tsx`.
2. Extraer helper de canciones externas.
3. Clarificar frontera `Songs` vs `Playlists`.
4. Validacion semantica de mappers y pads.
5. UX visible para errores de edicion inline.

## Veredicto

El proyecto esta funcional, empaqueta correctamente y ya tiene una primera capa de endurecimiento: escritura atomica, backups compartidos, tests Rust iniciales, Clippy limpio y CI. El gap principal restante es preservar XML desconocido y cerrar transacciones completas entre filesystem y `database.xml`.
